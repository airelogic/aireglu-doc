    Batch Invocation

    Asynchronous batch invocation of AireGlu Endpoints

    May 24, 2023

    Introduction

    It is possible to invoke an endpoint multiple times by using AireGlu’s, asynchronous, batch processing functionality.

    Simplified flow

    • A client makes a request to the AireGlu’s Async API via /Api/batch/{username}/{endpointName}/{environmentOrVersion}
      • This payload will contain the name of the endpoint to be batch invoked, the data to be passed into the endpoint and a callback endpoint to be invoked for each endpoint in the batch that has been processed.
    • The Async API passes the request to AireGlu’s Batch Event Handler. The service splits the batch into individual endpoint invocation messages and places each one onto a message queue.
    • AireGlu’s Runtime Service consumes the messages from the queue and invokes each endpoint
    • The result of each endpoint invocation is then placed onto a result queue
    • the Batch Event Handler reads the result of each endpoint invocation and triggers the invocation of the callback endpoint

    This diagram shows a simplified view of AireGlu’s asynchonous architecture.

    Access to this feature

    Your tenant/username must be added to the ALLOWED_ASYNC_USERS environment variable, you can add multiple users as a semicolon separated list. ENABLE_ASYNC should be set to true for your AireGlu deployment. You need to add MESSAGE_QUEUE environment variable indicating which message broker you are using. AireGlu currently provides first class support for RabbitMQ and Azure Service Bus (Amazon MQ for RabbitMQ is also compatible). Depending on which message broker you are using, you will also need to supply the message broker connection string. ConnectionStrings__AzureServiceBus / ConnectionStrings__RabbitMq

    Here is a further breakdown of what the enviroment variables:

    • ENABLE_ASYNC=true the runtime service will read this variable and subscribe to the necessary queues
    • ALLOWED_ASYNC_USERS determines which tenant/user will be allowed to use the batch invocation feature, this should be supplied as a semicolon separated list
    • MESSAGE_QUEUE indicates which message broker you are using, the options are RabbitMq or AzureServiceBus (if using AmazonMQ for RabbitMQ leave it set to RabbitMq)
    • Depending on which broker you have chosen you will also need to provide a connection string:
      • ConnectionStrings__AzureServiceBus
      • ConnectionStrings__RabbitMq if using Amazon MQ for RabbitMQ use this variable
    • USE_SEPARATE_EXECUTOR_SERVICES=false should be set to false for performance reasons

    How to invoke

    There are two methods of interacting with the Async Api Service: swagger or via a http client (e.g. Postman). The following example, demonstrates the process using Swagger.

    In this example we are performing batched invocations of an endpoint named my-batch-endpoint which is deployed to the production environment. We also provide an optional transaction id (if none is provided AireGlu will automatically create one).

    The request body is a JSON object containing:

    • the name of the callback endpoint to be invoked (my-callback)
    • request array containing the required input to be passed into the batched endpoint invocations (in this example my-batch-endpoint will be invoked twice with the input { "testInput": "hello" } and { "testInput": "world" })
    • environment or version of the my-callback-endpoint endpoint

    With a valid payload, the Async Api Endpoint will respond with a 202 Accepted.

    How authentication works

    The endpoint to be invoked via batch, should be authenticated as you would normally for any other endpoint. The callback endpoint will be run by the batch functionality regardless of its authentication status, however, it should have authentication set so that it is protected from someone running it directly.

    How will I monitor the requests?

    Usage data will be populated but it will be very difficult to follow. We strongly recommend ensuring that debug mode is turned off for the batch invoke and callback endpoints to minimise the amount of usage data (transaction data) stored when an endpoint is batch-invoked. The callback endpoint can be used to collect information about what the outcome of each invocation was. When calling the Async API you have the option to supply a batch transaction id (if none is supplied an id is automatically generated), the transaction id is made available in the input of the callback endpoint.

    What issues might I see?

    Depending on the number of batch invocations and complexity of the endpoint being batch invoked you may experience slow down to downstream services. We, therefore, recommend that you keep batch and callback endpoints, “light weight” (e.g light on data processing and number of tasks).

    Queues

    On startup the Message Broker Client will set up all the necessary queues and subscribe to them. The queues used for batch invocation are the following:

    • batch event
    • endpoint invoke
    • batch result
    • batch callback result

    In addition, each queue will have a corrosponding dead-letter queue set up by default, in the event that a message is unable to be proccessed it will enter the dead-leatter queue. You will need to have access to your message broker to view and handle these messages.

    Example endpoint definitions

    Here is an example endpoint definition that you can use to experiment with.

    Batch endpoint

    Versions:
- Input:
  Method: POST
  InputMode: JSON
  InputSource: Body
  Schema: >-
	{
   	 "$schema": "http://json-schema.org/draft-04/schema#",
   	 "description": "",
   	 "type": "object",
   	 "properties": {
   		 "test": {
   			 "type": "string",
   			 "minLength": 1
   		 }
   	 },
   	 "required": [
   		 "test"
   	 ]
	}
  XmlSchemas:
  ErrorEmail: ''
  StopOnTaskFailure: true
  Tasks:
  - Behaviour: Response
	RequestContext:
	ResponseHandlerContext:
	EmailContext:
	FaxContext:
	MeshContext:
	SmsContext:
	JobContext:
	ResponseContext:
  	OutputMode: JSON
  	MapType: Path
  	XsltParameters: []
  	StatusCode: 200
  	RetainStatusCode: false
  	Template: >-
    	{
   		 "test": "--*test*--"
    	}
  	Paths:
    	test: $.test
  	PathOrigins: {}
  	Functions: {}
  	FunctionLocations:
	RedirectContext:
	MappingContext:
	PdfContext:
	AdapterContext:
	SftpContext:
	JwtContext:
	Tests: []
	Filters: []
	Label: ''
	DataOrigin: -1
	UseCRLFLineEndings: false
  AuthenticationContexts: []
  Version: 1
  Active: true
  PersistTransactionData: false
Name: batch-test
Environments:
  Staging: 1
  Production: 1
